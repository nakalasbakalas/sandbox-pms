# Hosted Payment Cutover Runbook

This runbook is the production handoff for deposit-only hosted payments on Sandbox Hotel.

The intended live boundary is:

- `https://www.sandboxhotel.com` for brochure traffic only
- `https://book.sandboxhotel.com` for all guest booking and payment flows
- `https://staff.sandboxhotel.com` for staff PMS only

## Payment Boundary

Hosted checkout remains the only card-handling path.

The PMS:

- creates deposit payment requests
- generates hosted checkout sessions
- redirects guests to the provider-hosted page
- receives guests back on the `book` origin
- receives provider webhooks on the `book` origin
- applies confirmed deposits to the folio ledger

The PMS does not:

- collect raw card numbers
- store CVV or PAN
- render a custom card-entry form

## Required Production Environment

Set these in Render for the PMS service:

- `APP_ENV=production`
- `MARKETING_SITE_URL=https://www.sandboxhotel.com`
- `BOOKING_ENGINE_URL=https://book.sandboxhotel.com`
- `STAFF_APP_URL=https://staff.sandboxhotel.com`
- `APP_BASE_URL=https://book.sandboxhotel.com`
- `FORCE_HTTPS=1`
- `ENFORCE_CANONICAL_HOSTS=1`
- `AUTH_COOKIE_SECURE=1`
- `SESSION_COOKIE_SECURE=1`
- `TRUSTED_HOSTS=book.sandboxhotel.com,staff.sandboxhotel.com,sandbox-hotel-pms-v43m.onrender.com`
- `MAIL_FROM=reservations@sandboxhotel.com`
- `SMTP_HOST`
- `SMTP_PORT=587`
- `SMTP_USERNAME`
- `SMTP_PASSWORD`
- `SMTP_USE_TLS=1`
- `PAYMENT_PROVIDER=stripe`
- `PAYMENT_LINK_TTL_MINUTES=60`
- `PAYMENT_WEBHOOK_TOLERANCE_SECONDS=300`
- `STRIPE_API_BASE=https://api.stripe.com`
- `STRIPE_SECRET_KEY`
- `STRIPE_WEBHOOK_SECRET`

Also keep the admin payment selector aligned:

- in Admin -> Payments, use `active provider = env` unless you intentionally want a DB override
- in Admin -> Payments, keep deposit collection enabled if hosted deposit requests should remain live

## Provider Endpoints

The current code uses these live URL shapes:

- guest payment entry:
  - `https://book.sandboxhotel.com/payments/request/<request_code>?reservation_code=<reservation_code>&token=<confirmation_token>`
- guest payment return:
  - `https://book.sandboxhotel.com/payments/return/<request_code>?reservation_code=<reservation_code>&token=<confirmation_token>`
- provider webhook:
  - `https://book.sandboxhotel.com/webhooks/payments/stripe`

Important distinction:

- success and cancel return URLs are generated by the PMS dynamically per checkout session
- Stripe webhook delivery is the fixed dashboard-side endpoint that must be configured explicitly

## Stripe Dashboard Steps

1. Confirm the PMS service is live on `book.sandboxhotel.com`.
2. In the Stripe dashboard, create or update one webhook endpoint:
   - URL: `https://book.sandboxhotel.com/webhooks/payments/stripe`
3. Subscribe the webhook to the checkout-session lifecycle used by the PMS:
   - `checkout.session.completed`
   - `checkout.session.async_payment_succeeded`
   - `checkout.session.async_payment_failed`
   - `checkout.session.expired`
4. Copy the resulting webhook signing secret into:
   - `STRIPE_WEBHOOK_SECRET`
5. Put the live secret API key into:
   - `STRIPE_SECRET_KEY`
6. Keep the provider selection in Render as:
   - `PAYMENT_PROVIDER=stripe`
7. Redeploy if Render does not restart automatically after secret changes.

Notes:

- The PMS passes `success_url` and `cancel_url` to Stripe when it creates each hosted checkout session.
- There is no valid reason to point Stripe returns at `www` or `staff`.
- The brochure site must never receive payment returns or webhooks.

## Post-Cutover Smoke Test

1. From the PMS or public confirmation flow, create a deposit request for a test reservation.
2. Confirm the stored payment link starts with the hosted provider URL, not the PMS origin.
3. Open the guest-facing request entry URL on `book` and confirm it redirects to hosted checkout.
4. Complete a test payment using the provider's test mode.
5. Confirm the guest return lands on:
   - `https://book.sandboxhotel.com/payments/return/...`
6. Confirm the payment request moves to `paid`.
7. Confirm the reservation folio shows the deposit received exactly once.
8. Confirm a duplicate webhook delivery does not create a second folio deposit line.
9. Confirm the payment success email and reservation/payment history update correctly.

## Failure Checks

If the guest return page works but payment stays pending:

- verify the Stripe webhook endpoint URL is exactly the `book` URL above
- verify `STRIPE_WEBHOOK_SECRET` matches the current endpoint signing secret
- verify webhook deliveries are not blocked by Cloudflare or an old hostname

If webhooks arrive but fail signature verification:

- rotate the webhook secret in Render to the current dashboard value
- ensure the endpoint is configured for the correct Stripe environment

If payment links generate but send guests to the wrong host:

- verify `APP_BASE_URL` and `BOOKING_ENGINE_URL` are both `https://book.sandboxhotel.com`
- verify Admin -> Payments has not overridden the provider unexpectedly

If staff reset links or operational links point to `book` instead of `staff`:

- verify `STAFF_APP_URL=https://staff.sandboxhotel.com`

## Operational Ownership

What lives in code:

- URL generation rules
- webhook verification logic
- idempotent payment-to-folio application
- hosted-checkout-only payment boundary

What lives in infrastructure and provider config:

- Render secrets
- Stripe API keys
- Stripe webhook endpoint registration
- DNS and TLS for `book` and `staff`

## Cutover Completion Criteria

Hosted payment cutover is complete when:

- `book.sandboxhotel.com` is live and healthy
- Stripe webhook delivery to `book` succeeds
- deposit requests generate hosted checkout links correctly
- guest returns land on `book`
- confirmed payments post exactly one deposit line to the folio
- staff can see deposit state in reservation/front-desk/cashier views
- no part of the flow uses the brochure origin or staff origin for guest payment handling
