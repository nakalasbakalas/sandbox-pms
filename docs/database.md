# Database Workflow

This app uses Prisma with PostgreSQL. The Prisma schema is in `prisma/schema.prisma`, migrations are in `prisma/migrations`, Prisma CLI config is in `prisma.config.ts`, and the seed entrypoint is `prisma/seed.ts`.

## Local Disposable Postgres

For Windows Docker installation and troubleshooting, see [docker-setup-windows.md](docker-setup-windows.md).

Start the local databases:

```bash
npm run db:up
```

Copy the local env template and set any local-only secrets:

```bash
cp .env.local.example .env
cp .env.local.example .env.local
npm run db:ready
npm test
```

Prisma and the Node server read `.env`; Vite-compatible local tooling can also read `.env.local`.

The compose stack creates:

- `sandbox_hotel_dev` for normal local runtime and migration checks.
- `sandbox_hotel_e2e` for database-mutating E2E.

Use these URLs locally:

```env
DATABASE_URL="postgresql://sandbox:sandbox@localhost:55432/sandbox_hotel_dev?schema=public"
E2E_DATABASE_URL="postgresql://sandbox:sandbox@localhost:55432/sandbox_hotel_e2e?schema=public"
ALLOW_DB_E2E=false
```

`DATABASE_URL` is for normal runtime, migrations, and seed. `E2E_DATABASE_URL` is only for disposable or staging E2E. Keep `ALLOW_DB_E2E=false` unless you are intentionally running mutating E2E against a safe database. Local Docker maps host port `55432` to PostgreSQL's container port `5432` to avoid conflicts with any existing local Postgres service.

## Prisma Commands

```bash
npm run db:generate
npm run db:migrate
npm run db:seed
npm run db:status
npm run db:doctor
npm run db:e2e:ready
npm run db:ready
```

`db:migrate` uses `prisma migrate deploy`; do not use `prisma migrate dev` in deployed, staging, or CI environments.

## Seed Modes

Set `SEED_MODE` explicitly when the target matters:

- `SEED_MODE=dev`: local reference data plus local room inventory.
- `SEED_MODE=e2e`: disposable E2E reference data plus local room inventory.
- `SEED_MODE=prod-safe`: production-safe property and room-type configuration only, plus optional explicit users when `SEED_USERS_JSON` is set, or a legacy bootstrap admin when `SEED_ADMIN_EMAIL` and a seed password/hash are explicitly set.

The seed is idempotent and uses upserts. It does not delete existing data. `prod-safe` mode does not create fake guests, reservations, payments, invoices, operational room inventory, or demo staff users.

Generate production secret material with:

```bash
npm run prod:credentials
```

The command writes `ops/production-credentials.local`, which is ignored by git. It does not generate temporary admin credentials by default. For multiple real users, use `SEED_USERS_JSON` with approved emails and `passwordHash` values generated by `npm run security:hash-password`.

## Database-Mutating E2E

Run normal non-mutating E2E/contracts with:

```bash
npm run test:e2e
```

Run database-mutating E2E only with an explicit disposable or staging URL:

```bash
ALLOW_DB_E2E=true E2E_DATABASE_URL="postgresql://sandbox:sandbox@localhost:55432/sandbox_hotel_e2e?schema=public" npm run db:e2e:ready
ALLOW_DB_E2E=true E2E_DATABASE_URL="postgresql://sandbox:sandbox@localhost:55432/sandbox_hotel_e2e?schema=public" npm run test:e2e:db
```

PowerShell equivalent:

```powershell
$env:ALLOW_DB_E2E = 'true'
$env:E2E_DATABASE_URL = 'postgresql://sandbox:sandbox@localhost:55432/sandbox_hotel_e2e?schema=public'
npm run db:e2e:ready
npm run test:e2e:db
```

The guarded command refuses to run unless:

- `E2E_DATABASE_URL` is present.
- `ALLOW_DB_E2E=true` is present.
- `E2E_DATABASE_URL` is not the same as `DATABASE_URL`.
- The target URL does not contain production-like markers such as `prod`, `production`, `live`, `sandbox-hotel-pms-db-v43m`, or `sandbox_hotel_pms`.
- The target URL clearly contains a disposable marker such as `e2e`, `test`, `staging`, `ci`, `dev`, `local`, or `disposable`.

Before the mutating workflow runs, the script applies Prisma migrations to `E2E_DATABASE_URL` and runs the seed with `SEED_MODE=e2e`.

## Hosted Disposable/Staging Fallback

If Docker is unavailable, use a disposable hosted PostgreSQL database or a staging database that can be reset independently of production:

```env
E2E_DATABASE_URL="postgresql://USER:PASSWORD@HOST:PORT/DB?schema=public"
ALLOW_DB_E2E=true
SEED_MODE=e2e
```

Then run:

```bash
npm run db:doctor
npm run db:e2e:ready
npm run test:e2e:db
```

The database name, host, or user must clearly identify it as `e2e`, `test`, `staging`, `ci`, `dev`, `local`, or `disposable`; production-like targets are blocked.

## Production Room Onboarding

Production seed intentionally does not create operational room inventory. Import real approved rooms separately:

```bash
npm run rooms:import -- --file ./ops/rooms.production.json --confirm
```

In production or against production-like database URLs, the import also requires:

```env
ALLOW_PROD_ROOM_ONBOARDING=true
```

The import validates the property, room types, room numbers, floors, operational statuses, and housekeeping statuses before writing. It upserts by property and room number, and it does not create bookings, guests, invoices, payments, or demo operational data. Use `ops/rooms.example.json` as the file shape.

## Render Deployment

Render production provides `DATABASE_URL` through the managed PostgreSQL binding in `render.yaml`. The production web service should be connected to `https://github.com/nakalasbakalas/sandbox-pms`.

The Blueprint deploy path is:

```bash
npm ci --include=dev && npm run db:generate && npm run build
npm run db:migrate && npm run db:seed
npm run start
```

`SEED_MODE=prod-safe` is set in `render.yaml`, so production seed remains limited to safe configuration and explicitly configured real users or a legacy bootstrap admin. Mutating E2E must never use the Render production database URL.

Render skips `preDeployCommand` on free web service instance types. If production is running on the free instance type, apply migrations and seed from inside Render with a reviewed one-time build command such as `npm ci --include=dev && npm run db:generate && npm run db:migrate && npm run db:seed && npm run build`, then restore the normal build command. Prefer moving production to the committed `starter` plan before relying on automatic pre-deploy behavior.

## Final Launch Gates

Run before launch against the intended staging/disposable validation database:

```bash
npm run typecheck
npm run lint
npm test
npm run build
npm run db:doctor
npm run db:e2e:ready
npm run test:e2e:db
```

Production launch also requires:

- Render `DATABASE_URL` is provided by the managed database binding.
- Render `SESSION_SECRET` is generated from `render.yaml` or set from a generated secret bundle.
- Render `preDeployCommand` uses `npm run db:migrate`, which runs `prisma migrate deploy`, or an explicitly reviewed one-time Render build command has applied migrations when the service is on a free instance type.
- `npm run prod:preflight` passes against the exact production env values before deploy.
- No mutating E2E command uses the production database.
- Production seed uses `SEED_MODE=prod-safe`.
- Production room inventory is imported through `npm run rooms:import -- --file ./ops/rooms.production.json --confirm` with `ALLOW_PROD_ROOM_ONBOARDING=true`.
